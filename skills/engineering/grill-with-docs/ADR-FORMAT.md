# ADR 格式

ADR 位于 `docs/adr/`，使用连续编号：`0001-slug.md`、`0002-slug.md` 等。

按需创建 `docs/adr/` 目录：只有第一个 ADR 需要时才创建。

## 模板

```md
# {决策的短标题}

{1-3 句话：上下文是什么，我们决定了什么，以及为什么。}
```

就这些。ADR 可以只有一个段落。价值在于记录“做过这个决策”以及“为什么”，而不是填满各个小节。

## 可选小节

只有真正增加价值时才包含这些小节。大多数 ADR 不需要。

- **Status** frontmatter（`proposed | accepted | deprecated | superseded by ADR-NNNN`）：决策被重新审视时有用
- **Considered Options**：只有被拒绝的替代方案值得记住时才写
- **Consequences**：只有需要指出非显而易见的下游影响时才写

## 编号

扫描 `docs/adr/` 中已有的最高编号，然后加一。

## 何时提议 ADR

以下三点必须都成立：

1. **难以逆转**：未来改变主意的成本有意义
2. **没有上下文会令人意外**：未来读者看到代码会疑惑“他们到底为什么这样做？”
3. **真实权衡的结果**：确实存在可选方案，而你们基于具体理由选择了其中一个

如果某个决策容易逆转，就跳过；你们到时直接逆转即可。如果它不意外，就没人会问为什么。如果没有真实替代方案，就除了“我们做了显而易见的事”以外没什么可记录。

### 哪些算数

- **架构形状。** “我们使用 monorepo。”“写模型是 event-sourced，读模型投影到 Postgres。”
- **contexts 之间的集成模式。** “Ordering 和 Billing 通过 domain events 通信，而不是同步 HTTP。”
- **带锁定成本的技术选择。** 数据库、message bus、auth provider、部署目标。不是每个库都算；只记录那些要花一个季度才能换掉的东西。
- **边界和范围决策。** “Customer 数据由 Customer context 拥有；其他 contexts 只能用 ID 引用。” 明确的 no 和 yes 一样有价值。
- **刻意偏离显而易见路径。** “我们因为 X 使用手写 SQL 而不是 ORM。” 任何合理读者会默认相反选择的地方都算。这些能阻止下一个工程师把刻意选择“修掉”。
- **代码里不可见的约束。** “由于合规要求，我们不能使用 AWS。”“由于 partner API contract，响应时间必须低于 200ms。”
- **拒绝理由不显然的替代方案。** 如果你们考虑过 GraphQL，但因为细微理由选择 REST，就记录下来；否则六个月后还会有人再次建议 GraphQL。
