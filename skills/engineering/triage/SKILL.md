# 分诊（Triage）

让 project issue tracker 上的 issues 经过一个小型 triage roles 状态机。

Triage 期间发布到 issue tracker 的每条 comment 或 issue **必须**以此 disclaimer 开头：

```
> *This was generated by AI during triage.*
```

## 参考文档

- [AGENT-BRIEF.md](AGENT-BRIEF.md)：如何写 durable agent briefs
- [OUT-OF-SCOPE.md](OUT-OF-SCOPE.md)：`.out-of-scope/` knowledge base 如何工作

## Roles

两个 **category** roles：

- `bug`：某处坏了
- `enhancement`：新功能或改进

五个 **state** roles：

- `needs-triage`：maintainer 需要评估
- `needs-info`：等待 reporter 提供更多信息
- `ready-for-agent`：已完整指定，可交给 AFK agent
- `ready-for-human`：需要人类实现
- `wontfix`：不会处理

每个已 triaged issue 应该恰好带一个 category role 和一个 state role。如果 state roles 冲突，先标记并询问 maintainer，再做其他事。

这些是 canonical role names；issue tracker 中实际 label strings 可能不同。映射应该已经提供；如果没有，运行 `/setup-matt-pocock-skills`。

State transitions：unlabeled issue 通常先进入 `needs-triage`；然后再移动到 `needs-info`、`ready-for-agent`、`ready-for-human` 或 `wontfix`。Reporter 回复后，`needs-info` 返回 `needs-triage`。Maintainer 可以随时 override；遇到看起来异常的 transition，要先标记并询问再继续。

## 调用方式

Maintainer 调用 `/triage`，并用自然语言描述想要什么。解释请求并行动。示例：

- “Show me anything that needs my attention”
- “Let's look at #42”
- “Move #42 to ready-for-agent”
- “What's ready for agents to pick up?”

## 展示需要关注的内容

查询 issue tracker，展示三个 buckets，最旧优先：

1. **Unlabeled**：从未 triaged。
2. **`needs-triage`**：正在评估。
3. **自上次 triage notes 后有 reporter activity 的 `needs-info`**：需要重新评估。

展示 counts 和每个 issue 的一行摘要。让 maintainer 选择。

## Triage 具体 issue

1. **收集上下文。** 读取完整 issue（body、comments、labels、reporter、dates）。解析任何已有 triage notes，避免重新问已解决问题。使用项目领域术语表探索代码库，并尊重该区域 ADR。读取 `.out-of-scope/*.md`，并指出任何和此 issue 相似的历史 rejection。

2. **推荐。** 告诉 maintainer 你推荐的 category 和 state，并说明理由；同时给出和 issue 相关的简短 codebase summary。等待指示。

3. **复现（仅 bug）。** 在任何 grilling 前尝试复现：读取 reporter 的步骤，追踪相关代码，运行 tests 或 commands。报告发生了什么：成功复现并附 code path、复现失败，或细节不足（这是强 `needs-info` 信号）。确认复现会让 agent brief 强很多。

4. **Grill（如需要）。** 如果 issue 需要充实细节，运行 `/grill-with-docs` session。

5. **应用结果：**
   - `ready-for-agent`：发布 agent brief comment（[AGENT-BRIEF.md](AGENT-BRIEF.md)）。
   - `ready-for-human`：使用和 agent brief 相同的结构，但说明为什么不能委派（judgment calls、external access、design decisions、manual testing）。
   - `needs-info`：发布 triage notes（模板如下）。
   - `wontfix`（bug）：礼貌解释，然后关闭。
   - `wontfix`（enhancement）：写入 `.out-of-scope/`，在 comment 中链接它，然后关闭（[OUT-OF-SCOPE.md](OUT-OF-SCOPE.md)）。
   - `needs-triage`：应用该 role。如有部分进展，可选发 comment。

## 快速 state override

如果 maintainer 说 “move #42 to ready-for-agent”，信任他们并直接应用 role。确认你将要做什么（role changes、comment、close），然后行动。跳过 grilling。如果在没有 grilling session 的情况下移动到 `ready-for-agent`，询问是否要写 agent brief。

## Needs-info 模板

```markdown
## Triage Notes

**目前已确定：**

- 要点 1
- 要点 2

**还需要你提供（@reporter）：**

- 问题 1
- 问题 2
```

把 grilling 期间已解决的一切都记录到“目前已确定”下面，避免工作丢失。问题必须具体、可行动，而不是“请提供更多信息”。

## 恢复之前的 session

如果 issue 上已有 triage notes，读取它们，检查 reporter 是否回答了待解决问题，并在继续前展示更新后的情况。不要重新询问已解决问题。
